## Decisions

### Use VBS startup loop as non-admin fallback instead of requiring elevation

**Decision:** When `setup.ps1` detects no admin privileges, it installs a VBS script in the user's `shell:startup` folder instead of failing.
**Rationale:** The Windows test transcript showed the agent failing 5 times trying to work around the admin requirement. Most users won't run setup as admin. The VBS approach is silent (no console popup), auto-starts on login, and requires zero elevation.
**Alternatives considered:** Batch file in startup (shows console flash), PowerShell scheduled job (also needs admin on many configurations), user-level Task Scheduler via COM (inconsistent across Windows versions).

### Guard stat() with try/except rather than pre-checking existence

**Decision:** Wrap `stat()` in try/except rather than adding `if full_path.exists()` before it.
**Rationale:** EAFP is more robust on Windows — `exists()` can also fail on broken junctions, and there's a TOCTOU race. The except clause handles all failure modes in one place.
**Alternatives considered:** `full_path.exists()` check (still races, can also throw on broken junctions), `os.path.isfile()` (same issues).

## Lessons Learned

- **Pattern:** Windows file enumeration returns entries that can't be stat'd
- **Symptom:** `FileNotFoundError` or `OSError` on `Path.stat()` for files returned by `glob()` or `os.walk()`
- **Root cause:** NTFS junctions, broken symlinks, OneDrive cloud-only placeholders, and stale directory entries
- **Fix:** Always wrap `stat()` in try/except when iterating filesystem entries
- **Prevention:** Any file discovery code that calls `stat()` in a loop should have error handling

---

- **Pattern:** Windows Task Scheduler requires admin even for current-user tasks
- **Symptom:** `Access is denied` / `HRESULT 0x80070005` from `schtasks /Create` and `Register-ScheduledTask`
- **Root cause:** Default Windows security policy requires elevation for Task Scheduler operations
- **Fix:** Auto-detect admin with `WindowsPrincipal.IsInRole()` and provide a non-admin fallback
- **Prevention:** Never assume Task Scheduler access; always provide an alternative scheduling mechanism

---

- **Pattern:** Paths with spaces break Windows CLI tools
- **Symptom:** `schtasks` parses space-separated path segments as separate arguments
- **Root cause:** Insufficient quoting when interpolating paths into command strings
- **Fix:** Use proper XML escaping for template substitution, backtick-quoted strings for PowerShell commands
- **Prevention:** Always test with paths containing spaces; use structured APIs (XML, COM) over CLI string concatenation when possible
